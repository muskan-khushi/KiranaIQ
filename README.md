## The Problem

India has **13 million kirana stores** вҖ” the backbone of its вӮ№50 lakh crore retail economy. They are an enormous, largely untapped credit market for NBFCs. Yet underwriting them is nearly impossible through conventional means:

- вқҢ No ITR filings
- вқҢ No GST history
- вқҢ No balance sheets
- вқҢ No formal accounting

The only truth a kirana store has is the **physical world around it** вҖ” the shelves it stocks, the street it sits on, and the customers it serves.

**KiranaIQ solves exactly this.**

---

## What It Does

KiranaIQ is a full-stack AI underwriting platform. An NBFC field officer uploads 3вҖ“5 photos of a kirana store from their phone. In under 90 seconds, the system returns:

| Output | Description |
|--------|-------------|
| рҹ“Ҡ **Daily / Monthly Revenue Range** | Estimated with Вұ25% confidence bounds |
| рҹҸҰ **Loan Recommendation** | FOIR-based sizing with EMI range |
| рҹҡ© **Risk Flags** | 5 automated fraud tripwires |
| рҹ“Қ **Geo Intelligence** | Footfall, competition density, catchment analysis |
| рҹ”¬ **Feature Attribution** | Transparent, explainable signal breakdown |
| рҹ‘Ҙ **Peer Benchmarking** | Percentile rank vs. nearby comparable stores |

### Stage 1 вҖ” Vision Analysis
Each uploaded image is independently analysed by **Groq's multimodal LLM** (Llama 3.2 Vision). Outputs are then merged by `MultiImageMerger`, which computes a **consistency score** across images вҖ” high variance is a fraud signal.

Extracted signals per image:
- `shelf_density_index` (SDI) вҖ” fraction of shelf space stocked
- `sku_diversity` вҖ” estimated distinct product category count
- `dominant_categories` вҖ” mapped to Indian kirana price bands
- `refill_signal` вҖ” partially-empty shelves = recent high demand
- `store_area`, `image_type`, `image_quality`

### Stage 2 вҖ” Geo Intelligence
Three parallel OSM/Overpass queries (no paid API required):
- **Footfall Score (0вҖ“100)** вҖ” weighted sum of nearby POIs (bus stops, schools, markets) with distance decay
- **Competition Index (0вҖ“1)** вҖ” density of nearby grocery/kirana stores
- **Catchment Analysis** вҖ” residential density and income proxy from building/landuse data

All geo results are **Redis-cached for 6 hours** to avoid hammering the Overpass API.

### Stage 3 вҖ” Fraud Detection
Five automated tripwires, each with defined severity and recommended actions:

| Tripwire | Condition | Severity |
|----------|-----------|----------|
| `inventory_footfall_mismatch` | High SDI + very low footfall | рҹ”ҙ High |
| `low_image_consistency` | Consistency score < 0.55 | рҹ”ҙ High |
| `insufficient_image_coverage` | Missing interior / counter / exterior | рҹҹЎ Medium |
| `sku_geo_income_mismatch` | Premium SKUs in low-income catchment | рҹҹЎ Medium |
| `high_competition_low_sdi` | Dense competition + low shelf density | рҹҹў Low |

**Recommendation logic:** 2+ high-severity flags вҶ’ `reject`; 1 high-severity flag вҶ’ `needs_verification`; otherwise вҶ’ `approve`.

### Stage 4 вҖ” Fusion & Uncertainty
Revenue is estimated via **three independent paths**, fused by data-quality weights:

```
Path A (55%) вҖ” Working Capital Cycle:  Inventory Value Г— Daily Turnover
Path B (30%) вҖ” Geo-Demand Model:       Footfall Score вҶ’ benchmark range
Path C (15%) вҖ” Size-Based Model:       Revenue per sq ft (if size given)
```

The `UncertaintyEngine` computes confidence-adjusted ranges. Base spread is Вұ25%, **reduced** by strong signals (high SDI, known store size, multi-image consistency) and **widened** by fraud flags.

### Stage 5 вҖ” Loan Sizing + Peer Benchmarking
```
Max_Loan = Monthly_Net_Income Г— FOIR Г— Annuity_Factor(18% p.a., 18 months)
```
- FOIR = 0.45 (kirana industry standard)
- All loan amounts rounded to nearest вӮ№5,000
- Peer benchmarking via MongoDB `2dsphere` geo-query (2km radius)

## Quick Start

### Prerequisites
- Docker & Docker Compose
- A [Groq API key](https://console.groq.com) (free tier sufficient)

### 1. Clone & Configure

```bash
git clone https://github.com/your-org/kiranaiq.git
cd kiranaiq
cp backend/.env.example backend/.env
```

Edit `backend/.env`:
```env
GROQ_API_KEY=gsk_your_key_here
JWT_SECRET=your-secure-random-string-minimum-32-chars
MONGO_URI=mongodb://mongo:27017/kiranaiq
REDIS_URL=redis://redis:6379/0
```

### 2. Start the Stack

```bash
docker-compose up --build
```

This starts: FastAPI backend (port 8000), Celery worker (2 concurrent), MongoDB 7, Redis 7.

### 3. Seed Demo Data

```bash
docker-compose exec backend python scripts/seed_demo.py
```

Creates a demo NBFC officer account and a completed assessment with realistic mock results.

### 4. Explore the API

Open [http://localhost:8000/docs](http://localhost:8000/docs) вҖ” Swagger UI with all endpoints.

```bash
# Login
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "demo@kiranaiq.in", "password": "demo1234"}'

# Submit assessment (3вҖ“5 images required)
curl -X POST http://localhost:8000/api/v1/assess/ \
  -H "Authorization: Bearer <token>" \
  -F "images=@shop_front.jpg" \
  -F "images=@shelf_1.jpg" \
  -F "images=@counter.jpg" \
  -F "lat=28.6562" \
  -F "lng=77.2310"
```

## Key Design Decisions

**Why Groq instead of OpenAI Vision?**
Groq's LPU inference delivers ~10Г— lower latency at comparable quality for structured extraction tasks. For a field officer waiting on-site, response time matters enormously.

**Why OSM/Overpass instead of Google Maps?**
Zero cost at any scale. Overpass returns richer POI data for Indian cities than the Places API free tier. Results are cached in Redis so production load doesn't exceed rate limits.

**Why three estimation paths?**
Each path corrects the other's blind spots. The working capital path is accurate for well-stocked stores but fails on new inventory. The geo path is accurate for established markets but wrong for niche neighbourhoods. The size path anchors both. Weighted fusion is more robust than any single path.

**Why MongoDB over PostgreSQL?**
Assessment results are deeply nested, schema-free JSON documents. The `2dsphere` geo index for peer benchmarking is native. Schema flexibility lets the AI pipeline add new fields as the model evolves вҖ” no migrations.

## Production Deployment Notes

1. **JWT Secret** вҖ” Generate a cryptographically random secret: `openssl rand -hex 32`
2. **MongoDB Indexes** вҖ” Created automatically on startup. Ensure Atlas M10+ for `2dsphere` performance.
3. **Celery Concurrency** вҖ” Set `--concurrency` to match your Groq API rate limit (default: 2).
4. **Redis Persistence** вҖ” Use `appendonly yes` in Redis config to survive worker restarts.
5. **Image Storage** вҖ” Set `STORAGE_BACKEND=s3` and configure AWS credentials for production.
6. **CORS** вҖ” Update `allow_origins` in `main.py` to your production frontend domain.
